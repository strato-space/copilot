**Дистилляция предложений другой LLM**
- Сильная часть: одноразовый `bootstrap` внешних источников (repo/Figma) в локальные артефакты и работа дальше по локальным манифестам.
- Сильная часть: декомпозиция в DAG подзадач с явными зависимостями и параллельным запуском только unblocked-веток.
- Сильная часть: заранее “упакованный” контекст для воркера (цель, файлы, критерии приемки, валидация), чтобы не тратить ход на разведку.
- Слабая часть: попытка держать общий runtime-state в `docs/codex/state.json` как центральный источник истины.
- Слабая часть: чрезмерный фокус на “экономии токенов” как главной проблеме, а не на изоляции ролей и анти-байас механике.

**Критика с опорой на вашу позицию**
- Вы правы: при пустом контексте дочерних воркеров центральный `state.json` превращается в хрупкий shared-state (гонки, рассинхрон, ложная “актуальность”).
- Вы правы: корень не в токенах как таковых, а в смешении ролей и загрязнении контекста. Это и дает bias/дрейф решений.
- Тезис “не сбрасывать контекст оркестратора” полезен ограниченно; в вашем кейсе он легко перегревает parent-контекст и тащит лишние допущения в новые решения.
- “Логировать всё” не лечит качество экранных решений, если нет нормализованной карточки экрана и строгого role contract.
- Для UI-задач критично не “больше памяти”, а “лучше границы”: кто что видит, что может менять, какой формат handoff.

**Что оставить, что заменить**
- Убрать: `worklog.md` как отдельный журнал (дублирует `bd` и создает риск рассинхрона).
- Оставить: `bd` issue history/comments/events как единственный append-only журнал выполнения.
- Заменить: `state.json` на артефактный протокол handoff между ролями.
- Оставить: bootstrap-манифесты источников.
- Заменить: “скан репо” на task packet с whitelist путей.

**Практичный минимальный протокол (без овердизайна)**
- Единица работ: `Screen × Layer × Role` (например: `ScreenCard-12 × UI-Assembly × Router`).
- Вход воркера: `RoleCard + TaskPacket + AllowedPaths + AcceptanceCriteria`.
- Выход воркера: `Handoff` фиксированного формата (`changes`, `decisions`, `risks`, `next_role_input`).
- Оркестратор хранит только DAG и очередь, а не “весь мир” в контексте.
- Проверка качества: отдельный critic-role перед интеграцией, а не self-check того же воркера.
- BD Issues использовать как внешний план-граф, а не как “память” воркера.

## BD Issues: Forward-Only Graph без дублирования функционала

### Что уже умеет `bd` и не надо дублировать текстом
- Граф зависимостей: `bd dep add`, `bd dep tree`, `bd graph`.
- Выбор готового фронтира: `bd ready` (исключает `in_progress`, `blocked`, `deferred`, hooked).
- Просмотр блокеров: `bd blocked`.
- Статусы и фильтрация: `bd list --status ...`, `bd query "..."`.
- Закрытие + подсветка разблокированных задач: `bd close --suggest-next`.

### Правило контекста для воркеров
- Воркеры получают только `ready`-узлы и их прямые активные зависимости.
- Закрытые задачи не отправляются в воркер как полный контент.
- Из `closed` передается только краткий `closure receipt`: решение, артефакты, ограничения.
- Это и есть практический способ избежать перегрева: закрытые узлы не грузят prompt, грузят только артефактные ссылки.

### Операционный шаблон (оркестратор)
- `bd ready --json --limit N` для отбора unblocked задач.
- `bd dep tree <id> --direction=down --status=open` для активных блокеров.
- `bd show <id> --json` для task packet.
- После выполнения: `bd close <id> --reason "Done" --suggest-next`.

## Иерархическая модель проектирования

### Каноническая вложенность
- `CJM -> BPMN -> UserFlow -> Screens -> Widgets -> Atoms/Tokens`.
- Эта цепочка должна быть явно зафиксирована в артефактах проекта как структурный контракт.

### Роли, закрепленные за слоями
- Роль определяется уровнем абстракции, с которым она работает в цепочке `CJM -> BPMN -> UserFlow -> Screens -> Widgets -> Atoms/Tokens`.
- Роль не должна "прыгать" через уровни и смешивать ответственность соседних слоев.
- Делегирование идет вниз по слоям: верхний слой формулирует задачу следующему слою как контрактный handoff.
- Приемка идет вверх по слоям: роль-инициатор верифицирует результат нижележащего агента по своим критериям приемки.
- Это формирует иерархию управления: планирование и рамки сверху, реализация и детализация снизу, верификация обратно сверху.

### Явные определения сущностей
- Каждая сущность (`CJM`, `BPMN`, `UserFlow`, `Screen`, `Widget`, `Atom`, `Token`) должна иметь явный ID и тип.
- Для каждой сущности обязательно хранить `parent_id` и список `children_ids`.
- Нельзя оставлять неявную вложенность "по договоренности": связи должны быть машиночитаемыми.

### Обязательные связи
- Перекрестные ссылки между сущностями одного уровня и соседних уровней (например, экран <-> соседний экран в потоке, widget <-> reusable widget).
- Ссылки на доменную модель предметной области.
- Для каждого widget фиксировать: какой объект/коллекцию он отображает.
- Для каждого widget фиксировать: какие поля он использует.
- Для каждого widget фиксировать: из какого источника данных он читает.
- Для каждого widget фиксировать: какое действие/событие пишет обратно в домен.

### Практический смысл для оркестрации
- Воркеры получают контекст по точке в иерархии, а не "весь проект".
- Это снижает drift: роль работает в четко ограниченном поддереве.
- Это упрощает handoff: каждая роль передает изменения с привязкой к ID сущностей и доменных ссылок.

## Архитектурно-релевантные выводы из Steve Yegge (2026-01-19)

- Оркестратор становится главным инструментом разработки: не "один супер-агент", а система, где агенты управляют агентами.
- Разделение "worker vs factory" следует держать явным в архитектуре: роли исполнения и роли координации не смешивать.
- Модель "work as primitive" практичнее для масштабирования, чем модель "orchestration as primitive": сначала нормализованная единица работы, затем форма оркестрации.
- Аудитный data trail по work-item'ам (кто/когда/что сделал) дает долгоживущую устойчивость системы и снижает хаос при росте числа воркеров.
- Контуры качества должны быть отдельным слоем (review/loop/critic), а не растворены внутри каждого воркера.
- Форма оркестрации должна быть конфигурируемой (DSL/SDK и кастомные роли), а не жестко зашитой в код.
- Большие задачи нельзя отдавать "как есть": нужна штатная стратегия split/escalation/loop, иначе воркер упирается в контекст и деградирует по качеству.
- Для вашего кейса это напрямую усиливает тезис: центр архитектуры - `role separation + isolated context + контрактный handoff`, а не общий runtime-state.

Итог: у другой LLM полезные идеи по bootstrap и формализации задач, но архитектурный центр должен быть не `global state`, а `role separation + isolated context + контрактный handoff`.
